Slide 1

The Problem: A Gap No Tool Fills

Coaches Want Precision
Data-driven match predictions and tactical recommendations — not gut feelings.
LLMs Hallucinate Probabilities
Large language models reason fluently in natural language but guess probabilities — they do not
compute them.
Formal Methods Require Expertise
PAT computes exact probabilities but demands deep domain knowledge to write valid PCSP#
models by hand.
The Gap
No system bridges a natural-language coaching query to a formally verified, exact probabilistic
answer.

Slide 2

Our Approach: LLM Plans, PAT Verifies

Core Insight
Combine Gemini as a natural-language planner with
PAT (Process Analysis Toolkit) as a deterministic
verifier.
Result: Natural language in → formally
verified probability out.
LLM Role
Decides which tools to call — never generates probabilities.
PAT Role
Computes exact match win probabilities via PCSP# model checking.

Slide 3

Why Badminton?

Ideal for Markov Decision Processes (MDP)
Turn-based, rally-scoring mechanics create clear, discrete state transitions (e.g., Clear -> Drop -> Smash)
Rich, High-Fidelity Data
Open-source datasets (e.g., ShuttleSet) provide exact X/Y coordinates and granular shot semantics.
Complex Strategic Search Space
Players constantly adjust both shot selection and spatial targeting, creating a multi-variable optimization problem.

Slide 5

Dataset

ShuttleSet (KDD 2023)
Dataset Scope
44 professional matches · 3,685 rallies · 36,492 strokes · 27 players 
18 stroke types mapped to attack / neutral / safe categories 
(e.g., smash → attack,           lob → safe, drive → neutral).
Per Player Statistics
Serve mix: short serve rate vs. flick serve rate (must sum to 1.0)
Rally style: attack / neutral / safe ratios (must sum to 1.0)
Base win rates: serve-point win rate, receive-point win rate

Slide 6
PCSP# Template / Modelling

States
Score (points, games) + server identity (A or B)
Transitions
Serve → rally outcome (A wins point or B wins point) with weighted probabilities
Scoring Rules
Standard badminton: play to 21, deuce to 30, best-of-3 games
Probabilities
Player stats + style-differential adjustment (influence weights via OLS regression) populate {{placeholder}} syntax. 
Values are injected as integer weights (×10,000) for pcase statements.

Slide 7
LLM Agent and Internal Tooling

Planner
Generates a JSON plan based on the query + system prompt
Task type, player names, tool-call sequence. 
Falls back to keyword heuristics if API is unavailable.
Agent Executor
Deterministically retrieves players stats → builds model → runs PAT → summarizes results
No LLM in the critical verification path to avoid any randomness or bias
PAT Verifier
Exhaustive state-space exploration
Outputs an exact probability that Player A wins. 
Supports mock mode (logistic approximation) for development.

Slide 8

Core capabilities

Predict
Who is more likely to win?
Builds a single PCSP# model from both players' stats, runs PAT, and returns chances % of A winning as an exact probability
Strategy Optimization
How should Axelsen adjust to beat Momota?
Grid search over serve-mix and rally-style adjustments (±5%, ±10%, ±20%), bounded by L1 change ≤ 0.3
Returns the top-5 strategy alternatives ranked by win probability improvement

Slide 9

Key Engineering Decisions

Pydantic Strict Models
To enforce that all probability distributions sum to 1.0 
Data bugs are caught at construction time, not at runtime

OLS Influence Weights
Style-differential → point-share relationship estimated via OLS regression on match data

H2H Blending
Base rates are blended with H2H stats at up to 35% weight for improved accuracy for queries with prior history

Laplace Smoothing
Applied to win rates to prevent extreme probabilities arising from small sample sizes in the dataset
